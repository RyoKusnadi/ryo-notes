# Lecture Notes: Building a Coding Agent From Scratch
## 1. Core Terminology
To understand how an agent works, you must understand the three types of messages (prompts) that make up the conversation history sent to the Large Language Model (LLM):
*   **System Prompt:** The foundational instructions. It defines the AI's overall behavior, its role, its boundaries, what tools it has access to, and what it is strictly forbidden to do.
*   **User Prompt:** The specific, custom requests or inputs provided by the human.
*   **Assistant Prompt:** The actual responses, text, or tool-call requests generated by the LLM.

## 2. The Fundamental Architecture: "The Loop"
At its core, a coding agent is not a magical black box; it is a continuous loop of communication between three entities: **You (the User)**, **The LLM**, and **The Tools (your local environment)**.

1.  You send a request.
2.  The LLM receives the request along with the System Prompt.
3.  Instead of just replying with text, the LLM can decide to use a tool (e.g., "I need to read a file first").
4.  Your local program intercepts this request, executes the tool offline (on your actual computer), and gets the result.
5.  The result is fed back to the LLM as a new message.
6.  The LLM evaluates the result and decides: *Do I need to use another tool, or am I ready to reply to the user?*
7.  This repeats until the LLM decides the task is complete and generates a final text response for you.

## 3. Step-by-Step Implementation (Terminal Agent)
The lecture builds a simplified agent that runs in the terminal (not a full graphical IDE). It is given three specific tools to interact with your local files:
*   `read_file`: Reads the contents of a specific file.
*   `list_dir`: Shows the contents of a specific folder.
*   `edit_file`: Creates a new file or modifies an existing one.

### The Setup
*   **Environment:** Uses `uv sync` to install dependencies.
*   **Authentication:** Requires an OpenAI API key. If not set in the environment variables, the code prompts you to paste it directly into the terminal.
*   **Model Configuration:** Uses a model (like GPT-5 in the demo) with the token limit restriction removed to allow for longer, uninterrupted interactions.

### The Code Structure (The Two Loops)
The main logic of the agent is built on nested loops:

**Outer Loop (The Human Interaction Loop):**
*   Waits for you to type a message in the terminal.
*   Adds your message to the conversation history as a "User" role.
*   Triggers the Inner Loop.
*   Once the Inner Loop finishes (meaning the AI has given you a final answer), it prints the AI's response and waits for your next message.

**Inner Loop (The AI & Tool Execution Loop):**
*   Sends the entire conversation history to the LLM.
*   Receives the LLM's response.
*   **Checks for Tool Use:** It parses the AI's response to see if it contains a command to use a tool (e.g., extracting a JSON payload that specifies `{"tool": "read_file", "path": "main.py"}`).
    *   *If a tool is requested:* The program executes the tool locally, takes the result, adds it to the conversation history, and loops back to ask the LLM what to do next.
    *   *If NO tool is requested:* This means the AI is done working and is just talking to you. The program adds this text to the history as an "Assistant" role, breaks out of the Inner Loop, and returns control to the Outer Loop.

## 4. The "Secret Sauce": How Production Agents (like Claude Code) Optimize
Building a basic loop is easy, but making an agent reliable, fast, and capable of handling massive codebases requires advanced prompt engineering and architectural tricks. The lecture breaks down the underlying design of top-tier agents like Claude:

### A. Front-loading Context with Targeted Prompts
Instead of giving the AI a massive, vague document, production agents use highly specific, concise prompts at the very beginning of the System Prompt. This immediately locks the AI into the exact persona, rules, and context it needs, reducing confusion.

### B. System Reminders Everywhere (Preventing "Drift")
As a conversation gets longer, the AI tends to "drift" or forget its original instructions. To combat this, advanced agents inject hidden `<system-reminder>` tags throughout the entire conversation.
*   These reminders are placed not just in the System Prompt, but also inside User Prompts, Tool Calls, and Tool Results.
*   They act as constant, subtle nudges to keep the AI strictly aligned with its core directives, no matter how deep into the task it gets.

### C. Command Prefix Extraction
When an LLM outputs text, it doesn't natively "execute" code. The agent's software must parse the raw text output from the AI. It looks for specific prefixes or formats (like a specific JSON structure) to know exactly when the AI is done "thinking" and is requesting a tool. The software then extracts the parameters (like the file name) and executes them.

### D. Spawning Sub-Agents (Managing Context Overload)
When an AI reads too many files, the "context window" fills up. When the context gets too large, two things happen:
1.  The AI's accuracy drops (it gets confused by too much information).
2.  The processing speed slows down significantly.

**The Solution:** Sub-agents. Instead of forcing one AI to hold the entire massive project in its memory, the main agent acts as a coordinator. It spawns separate, smaller AI instances (sub-agents) to handle specific, isolated tasks (e.g., "You go read and fix this specific module"). Once the sub-agent finishes, it returns a short summary to the main agent. This keeps the context window clean, fast, and highly accurate.
